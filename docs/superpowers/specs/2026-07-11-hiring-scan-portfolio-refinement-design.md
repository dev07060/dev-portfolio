# 채용 스캔 중심 포트폴리오 개선 설계

## 목표

현재 채용형 포트폴리오의 기술적 신뢰는 유지하면서, 채용 담당자가 모바일과 데스크톱에서 핵심 역량·대표 근거·경력을 더 짧은 시간 안에 파악하도록 정보 중복과 세로 길이를 줄인다.

근거 문서는 `output/layout-audit-2026-07-11/audit.md`이며 사용자가 해당 권장 순서대로 구현하는 안을 승인했다.

## 최종 정보 구조

페이지 순서는 다음으로 고정한다.

1. 앱바
2. Hero, 공개 근거, 요약 역량 4개
3. 대표 기술 사례 3개
4. 경력 요약 6개
5. 추가 프로젝트 compact archive 5개
6. 채용 CTA와 공개용 이력서
7. Footer

`핵심 개발 역량`과 `보완 오픈소스`는 독립 섹션으로 렌더링하지 않는다.

## 1. 앱바

- 왼쪽 `DEV PORTFOLIO`는 유지하고 `#about`으로 연결한다.
- 같은 목적지의 `소개` 링크는 제거한다.
- `대표 기술 사례`는 앱바에서 `기술 사례`로 축약한다.
- 앱바 항목은 `기술 사례`, `경력`, `연락` 순서다.
- 각 내비게이션 링크는 최소 44px 높이의 터치 영역을 갖는다.
- 320px에서 앱바 내부 가로 스크롤과 잘린 링크가 없어야 한다.

## 2. Hero와 요약 역량

- 이름, 역할, 기술 포지션, 가치 제안, 공개 패키지, 총경력, CTA는 유지한다.
- 사용자 확정 브랜드 `DEV PORTFOLIO`를 제외한 장식용 영문은 제거한다.
- `Byeonghee Oh`는 Hero에서 제거한다.
- 기존 역량 4개는 독립 카드 섹션 대신 Hero 안의 2열 compact list로 배치한다.
- 각 역량은 제목과 근거 프로젝트명만 표시한다.
  - Flutter 제품화·릴리스 — Easy Contract Viewer
  - 온디바이스 Retrieval/RAG — mobile_rag_engine
  - Rust FFI·네이티브 검색 — mobile_rag_engine
  - 검색 백엔드·평가 운영 — Swifty-law

## 3. 대표 기술 사례와 공개 근거 등급

- 대표 순서 `mobile_rag_engine → Easy Contract Viewer → Swifty-law`는 유지한다.
- 외부 근거 링크가 있는 사례는 결과 라벨을 `공개 근거가 있는 결과`로 표시한다.
- 외부 근거 링크가 없는 사례는 결과 라벨을 `핵심 결과`로 표시한다.
- `검증 가능한 결과`라는 단일 라벨은 제거한다.
- `rag_engine_flutter 0.18.3`은 독립 오픈소스 섹션에서 제거하고 `mobile_rag_engine` 상세의 `관련 공개 패키지`로 통합한다.
- 관련 패키지에는 이름, 버전, 한 줄 관계 설명, pub.dev, GitHub, 기술 태그를 제공한다.

## 4. 경력 요약

- 6개 경력과 최신순 순서는 유지한다.
- 기본 상태에는 회사, 기간, 역할, 고용 형태 badge, 대표 성과 1개만 표시한다.
- 요약 설명, 두 번째 성과, 관련 프로젝트는 네이티브 `details` 안에 둔다.
- 고용 형태는 역할 문장과 분리해 좁은 화면 줄바꿈을 안정화한다.
- `검증 가능한 결과` 표현은 경력 설명에서 사용하지 않는다.

## 5. 추가 프로젝트 compact archive

- 추가 프로젝트 5개와 상세 모달 연결은 유지한다.
- 기존 이미지 중심 대형 카드 grid는 제거한다.
- 각 행은 번호, 프로젝트명, 유형, 담당 범위, 핵심 기술 최대 3개, `상세 보기` 버튼만 표시한다.
- 데스크톱은 한 행에 정보를 배치하고 모바일은 읽기 순서 그대로 세로 배치한다.

## 6. 프로젝트 상세와 성능

- 데스크톱의 텍스트·시각 자료 2열 구조는 유지한다.
- 모바일은 제목과 공개 근거 다음에 대표 이미지를 배치하고, 그 뒤에 문제·기여·결과를 배치한다.
- 첫 대표 사례의 첫 이미지는 eager/priority 로딩 대상으로 지정해 현재 LCP 경고를 제거한다.

## 7. 공개용 이력서와 CTA

- 원본 이력서의 전화번호, 주소, 희망연봉과 비공개 개인 정보는 공개하지 않는다.
- 사이트에 이미 공개한 이름, 이메일, GitHub, 역할, 기술, 경력, 프로젝트 근거만 사용해 별도 PDF를 생성한다.
- 공개 파일 경로는 `/oh-byeonghee-resume-ko.pdf`로 고정한다.
- Hero와 마지막 CTA의 `이력서 보기`가 같은 파일을 새 탭에서 연다.

## 8. 한국어 문구

- 유지 예외: `DEV PORTFOLIO`, 제품명, 라이브러리명, 기술 용어.
- 섹션 eyebrow는 `대표 흐름`, `경력`, `프로젝트`, `연락`으로 통일한다.
- `Trade-off` 표기는 `트레이드오프`로 바꾼다.

## 검증 기준

- 구조 테스트가 최종 섹션 순서, 앱바 항목, 독립 역량·오픈소스 섹션 부재, compact archive, 경력 details, 근거 라벨, 공개 이력서 링크를 검증한다.
- 320px에서 문서와 앱바에 가로 overflow가 없어야 한다.
- 390px 모바일 문서 높이는 기존 감사값 10,491px보다 유의미하게 짧아야 하며 목표는 8,000px 이하다.
- 데스크톱 1440px, 모바일 390px, 좁은 모바일 320px에서 Hero·대표 사례·경력·archive·CTA를 시각 확인한다.
- 프로젝트 상세는 데스크톱과 모바일에서 닫기, 스크롤, 대표 이미지 순서를 확인한다.
- `npm run test`, `npm run lint`, `npx tsc --noEmit --incremental false`, `npm run build`, Playwright 접근성 테스트가 모두 통과해야 한다.
- VoiceOver나 스크린리더 수동 검증은 범위에 포함하지 않는다.
